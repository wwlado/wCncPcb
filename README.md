In this article, I will cover the production of simple printed circuit boards (hereinafter referred to as PCBs) using a small CNC router and the wCncPcb application.

During practical teaching, I often need to produce multiple simple PCBs in a fast and efficient way. Initially, I designed my PCBs on graph paper and then manually wrote the gCode for the router. Later, I designed a simple application where I drew a sketch of the holes, slots, and contours, and the application then generated the gCode from it. This allowed me to get a board exactly according to my ideas. I named this application wCncPcb.

After some time, I acquired a small CNC router, the Route 3018 Pro (hereinafter referred to as the 3018). I decided to rewrite the entire application from scratch, fixing the shortcomings of the first version, adding component footprints, and introducing the ability to mill double-sided PCBs. The second version of wCncPcb generated gCode that I could send to a Proxxon mill using the ncCad9 application, or to the 3018 router using the gSender application. The small 3018 router also allows loading files via an SD card to manufacture the PCB offline. At that point in time, I preferred making PCBs on the large Proxxon mill, as the small 3018 router frequently "froze." In gSender, I could see the computer waiting for a task confirmation from the router, and conversely, the router waiting for the next command. The magic "continue" button was missing. There were even situations where the small router plunged all the way into the wasteboard or cut slots in completely wrong places.

On the other hand, the 3018 router is compact, handy, and inexpensive. Therefore, my priority was to fix the communication between the computer and this machine. Fortunately, extensive documentation is available for GRBL (the firmware interpreter used in the 3018), which helped me a lot. To eliminate all errors that could arise from poor communication, I decided to use a USB-to-USART cable connection instead of a standard USB cable connection to the PC. I connected the USART side to the router's communication connector intended for Bluetooth. I must humbly admit that the application now communicates with the router absolutely flawlessly. In my opinion, the original issue was caused by the power supply. The USB-to-USART cable supplies a backup 5V voltage to the mainboard, so if there is a voltage drop in the adapter, the communication does not fail. But that is just my personal opinion. Once the application was reliably communicating with the router, I added a few more functions to manage the GRBL interpreter.

Based on previous experience, I built the aforementioned "continue" button into wCncPcb just to be safe. It can be used in an emergency if the router fails to acknowledge data receipt and the process freezes at some point.

Since the communication was working so well, a question kept popping up in my mind: "What about trying Bluetooth?" I bought the cheapest Bluetooth (hereinafter referred to as BT) module available at the time – the JDY-33. I thought that if I set the serial port speed in the application to 115200, the module would communicate with the router at the same speed. That is a misconception. The BT communication speed must be configured via USART. Our application can help with that as well. The console includes custom buttons that reconfigure the BT module to the required speed. I also soldered a 1500 uF capacitor to the BT module's power supply due to the mentioned voltage fluctuations. It works really reliably. The biggest advantage of wireless communication is that the laptop can be in a different room than the router, preventing milling dust from getting into the computer.

I would like to point out one important feature. When sending data directly to the 3018, the gCode is generated progressively during the milling process. If we selected only two items to send, the application would run the initialization code, send the code for both items, and finish with the ending code. This makes it possible to fix poorly milled tracks or finish a product in case of a malfunction. It also provides another huge advantage: the ability to use surface height correction.

Surface unevenness was another thing that kept me up at night. At first, I leveled the wasteboard by placing small pieces of paper underneath it. Later, I thought of programming a "wasteboard leveling by the router" feature. It works simply: it mills the surface within the specified dimensions, making the milled part of the wasteboard 100% flat relative to the tool. However, the PCB itself is often warped, and during fine milling (for example, 0.3 mm), every slight bend is a major problem. This is where the need for surface unevenness measurement arose. GRBL has a special command that stops the motor when input A5 is triggered and sends the exact coordinates of the spindle in an error message. Since the PCB has a copper (conductive) surface, and the motor, coupler, and milling bit are also metallic (conductive), it is enough to connect one wire of the A5 input to the router motor and the other to the PCB. This feature is known in GRBL as PROBE. The wCncPcb application features three levels of surface probing:
1. **The simplest level:** It basically just sets the zero point when the bit touches the PCB.
2. **Four-point probing:** It performs four measurements at the edges of the PCB. This allows the code interpreter to create a Z-coordinate correction map.
3. **Multi-probing (The highest level):** It performs a grid of measurements, which is suitable for larger PCBs. I should add that multi-probing only makes sense if the tracks are short. I kept this in mind and implemented a feature in the application to split long tracks into shorter segments.

From the PCB design perspective, it is important to remember that the application shows the view from the component side, i.e., a top-down view (just like in Eagle). To build a simple circuit, you just need to place the necessary components and draw lines where the slots should be milled. The "contour" tool mills the outline of the PCB with holding tabs. For more complicated circuits, the board design is significantly more complex. For intricate circuits, it is better to design the PCB in a conventional application like Eagle and then import the data into wCncPcb. This led me to implement several import options. Creating the overall outline of connected tracks was initially problematic. Therefore, I first used the FlatCAM application for this purpose. I generated a GERBER file from Eagle and imported it into FlatCAM. Finally, I loaded the generated DXF outline file into wCncPcb. I should point out that DXF does not contain holes. Therefore, I had to program an Excellon import feature directly from GERBER. It seemed quite complicated to me, so I tried importing an image and creating the track outlines from it. It works reasonably well, but you need to maintain the correct DPI resolution and then align the image with the holes from the Excellon file. I wasn't fully satisfied with this approach; I wanted to import GERBER with a single click. 

This was ultimately solved by the Clipper library (authored by Angus Johnson), which can unify polygons. Finally, I was able to create polygons (tracks, pads, etc.) from the GERBER file, use `Clipper.union` to merge connected objects into a single shape (polygon), and in the final step, export the outline of the unified shape offset by the radius of the milling tool. Currently, the import works seamlessly: you just click on any GERBER file, and the application automatically searches for all the other necessary files. I tailored the decoding algorithm for DXF, GERBER, and EXCELLON files to match the existing files exported from EAGLE and FlatCAM. It is possible that another application might generate a GERBER file with a different structure, and the file might not load correctly. For example, a GERBER file from Eagle contains the "AMOC8" macro, which creates an octagon. The wCncPcb application does not create the "OC8" object using this macro but interprets it directly as an octagon object. If anyone imports a GERBER file that loads incorrectly, please write to me. I will be very happy to fix any bugs.

After importing a PCB board from Eagle, additional requirements emerged: thinner milled slots (requiring tool changes) and changing the tool speed or diameter. I programmed three more buttons that trigger individual phases of PCB production. The first button starts drilling small holes (up to the radius of the tool/bit), the second button mills only the slots (at a higher speed), and the last button starts milling larger holes (holes larger than the tool/bit radius) and contours.

Although we can use the router to create boards for SMD assembly, it didn't really catch on in practical teaching. The milled slot is too thin for beginners, and students prefer assembling boards that have significantly larger soldering pads and gaps. I returned to the idea of manufacturing PCBs using a single milling bit (e.g., 0.8 mm). Therefore, I added the option to change the milling speed depending on whether the tool is drilling, clearing copper (milling slots), or cutting (milling the outline).

The need to manufacture multiple boards at once gave me the opportunity to program a special button. After setting the desired number of PCBs, you just press it, and the router automatically measures the surface unevenness and mills the required number of boards using a single bit. Today, I can't recommend the 3018 router enough.

Finally, I would just like to add that wCncPcb is not a commercial, professional application. I programmed it as freeware for PCB production in school and home environments. Therefore, it is not bound by any traditional license (not even Open-Source). The application is primarily designed for the Windows operating system, but thanks to "wine," it also works under Linux.
